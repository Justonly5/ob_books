### 核心概念

跨域网络指应用跨越不同物理位置（机房/数据中心/地域/云厂商）进行通信，本质上是把原来同机房内 < 1ms 的调用，变成需要跨越物理距离的远程调用。

RTT：**Round Trip Time**，往返时延。直白理解

```
你发出一个网络请求，到收到对方响应，这段时间就是 RTT

A 机房应用  ──发请求──▶  B 机房服务
A 机房应用  ◀──返回-──   B 机房服务

RTT = 发出去 + 回来 的总时间
```

### 四大核心影响

#### ① 延迟放大（最直接）

```
同机房调用：< 1ms
同城不同机房：1~5ms（光纤）
跨城市专线：10~30ms（北京→上海）
跨国/跨云公网：50~200ms

链路放大效应：
  一次用户请求 → 调用 10 个内部服务
  每次跨域调用 30ms
  串行调用总耗时 = 10 × 30ms = 300ms
  原本同机房只需 10ms
```

Spring Boot 里的体现就是接口响应时间整体拉长，连接池 `connectionTimeout` 频繁触发。

#### ② 带宽瓶颈

```
专线带宽有上限（通常 1G/10G，费用极高）
业务高峰期流量突增 → 专线打满 → 丢包
大数据量传输（文件、图片、报表）尤其明显

典型场景：
  A 机房的报表服务查询 B 机房的数据库
  每次查询返回 10MB 数据
  100 并发 = 1GB/s，专线直接打满
```

#### ③ 数据一致性问题

```
主库在 A 机房，从库在 B 机房：
  写入主库 → 复制到从库有延迟（跨域复制延迟 50~500ms）
  A 机房写完立即在 B 机房读 → 可能读到旧数据

跨域分布式事务：
  2PC 需要多次跨域网络往返
  每次往返 30ms × N 次 = 事务耗时极长
  网络抖动导致事务超时率上升
```

#### ④ 故障传播与级联

```
专线中断（光缆被挖断是真实案例）：
  A 机房所有调用 B 机房的请求全部超时
  线程池被打满（等待超时响应）
  A 机房应用跟着崩溃（级联故障）
  
没有熔断的后果：
  1 个服务跨域调用超时 30s
  200 个线程全部阻塞在等待
  Tomcat 线程池耗尽，整个应用无响应
```

### 应用层的应对策略

#### 超时配置收紧


```yaml
# 跨域调用必须设置严格超时，不能用默认值
spring:
  data:
    redis:
      timeout: 500ms        # 跨域 Redis 超时从 2s 收到 500ms

# HTTP 客户端跨域调用
okhttp:
  connect-timeout: 3s
  read-timeout: 5s          # 根据跨域延迟基线设置，不要太宽松
```

#### 熔断降级（必须）



```java
// 跨域调用必须加熔断，防止级联故障
@CircuitBreaker(name = "cross-region-service",
    fallbackMethod = "fallback")
public Result callRemoteService(Request req) {
    return remoteClient.call(req);
}

public Result fallback(Request req, Exception e) {
    // 降级：返回缓存数据、默认值、或提示稍后重试
    return Result.degraded("服务暂时不可用，请稍后重试");
}
```

#### 数据就近原则

```
原则：读写尽量在同一机房完成，跨域只做数据同步

典型架构：
  A 机房用户 → 读写 A 机房数据库（本地优先）
  B 机房用户 → 读写 B 机房数据库（本地优先）
  A ↔ B 只做异步数据同步（允许短暂不一致）
```

#### 异步化跨域调用



```java
// 同步跨域调用：线程阻塞等待，放大延迟影响
Result result = crossRegionClient.call(req);  // 阻塞 30ms

// 异步跨域调用：不阻塞业务线程
// 通过消息队列解耦，A 机房发消息，B 机房异步消费
kafkaTemplate.send("cross-region-topic", event);
// 立即返回，B 机房消费者异步处理
```

### 风险汇总

|风险类型|触发条件|对应用的影响|应对方案|
|---|---|---|---|
|延迟放大|串行跨域调用|响应时间成倍增加|并行调用、异步化|
|带宽打满|大数据量传输|丢包、超时|数据压缩、分页、就近读|
|主从延迟|跨域数据库复制|读到脏数据|读主库、允许最终一致|
|专线中断|光缆故障、设备故障|级联崩溃|熔断、降级、多路由|
|网络抖动|流量高峰、设备异常|超时率上升|重试 + 幂等、超时收紧|
|安全暴露|数据跨网络传输|中间人攻击|TLS 加密、专线隔离|

跨域网络的核心思路是：**能不跨就不跨，必须跨就做好隔离和降级**，把跨域调用的失败当作正常情况来设计，而不是异常情况。
